# retrospectives/ — Engineering Retrospective Reports

> **Read-only observation reports** generated by the `meta-auditor` sub-agent, triggered by the `/meta-audit` command.
> After human review, adopted suggestions are hardened into code/rules/commands via the normal PR process.

## Naming Convention

```
<YYYY-MM-DD>-meta-audit.md
```

Running multiple times on the same day overwrites the report for that day. To preserve history, avoid re-running on the same day, or manually rename the old report.

## What a Report Is / Is Not

| Is | Is Not |
|----|--------|
| A snapshot observation at a point in time | A to-do list (do not check off items as "handled") |
| Suggestions and discussion | Mandatory requirements |
| A baseline for trend comparison | An editable document |
| Input for human decision-making | Grounds for automatic agent execution |

**Immutability principle**: Once generated, a report is never modified. The next `/meta-audit` run generates a new report based on the current state, making it easy to compare which issues were resolved and which remain.

## Process for Adopting Suggestions

1. **Review the report** — Open the latest report and go through the 🔴/🟡/🔵 sections
2. **Triage each suggestion**:
   - Adopt → proceed to next step
   - Reject → skip (the next report will surface it again if the issue persists)
   - Uncertain → open a GitHub issue or discuss with the team
3. **Harden via normal process**:
   | Suggestion Type | Action |
   |----------------|--------|
   | Rule inconsistency / rule drift | Edit `.claude/rules/` |
   | Command / skill / agent issue | Edit the corresponding `.md` file |
   | Code violation (hardcoding, etc.) | Use `/fix` or normal development flow |
   | Outdated documentation | Edit `docs/` or `CLAUDE.md` |
4. **Do not annotate the report** — The next `/meta-audit` run will naturally detect changes and reflect them in the "Trends · Resolved" section

## How to Trigger

### Manual

```bash
# Full scan
/meta-audit

# Scan a specific dimension only
/meta-audit --focus=traceability
/meta-audit --focus=dead-links
```

### Recommended Frequency

- **After milestones**: Run once after a major release or milestone to assess overall health
- **Regularly**: Every 1–2 weeks to prevent drift from accumulating
- **After adding new mechanisms**: Run once after adding a new command / rule / agent to check for consistency issues

**Not recommended**: Scheduled automation — reports no one reads are just noise; manual triggering ensures someone will actually read them.

## Design Principles

- **Observe, don't judge** — Reports are not absolute truth; they are observations based on current rules
- **Humans decide, not agents** — Agents change nothing; all modifications go through human review
- **Trend-based, not single-point** — The same issue appearing across multiple reports = genuinely needs fixing; appearing once may be noise
- **Preserve positive feedback** — Reports should also list "what's going well" to avoid a purely negative lens

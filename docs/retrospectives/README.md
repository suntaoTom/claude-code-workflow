# retrospectives/ — 工程复盘报告

> `meta-auditor` 子代理生成的**只读观察报告**, 由 `/meta-audit` 命令触发。
> 人类 review 后, 把采纳的建议走正常 PR 流程固化到代码/规则/命令里。

## 命名约定

```
<YYYY-MM-DD>-meta-audit.md
```

一天内多次跑会覆盖同日报告, 想保留历史就不要同日重跑, 或手动重命名旧报告。

## 报告是什么 / 不是什么

| 是 | 不是 |
|----|------|
| 某一时刻的快照观察 | 待办清单 (不要在报告里打勾标「已处理」) |
| 建议和讨论 | 强制要求 |
| 趋势对比的基准 | 可修改的文档 |
| 供人类决策的输入 | agent 自动执行的依据 |

**不可变原则**: 报告一旦生成, 不再修改。下次 `/meta-audit` 跑时会基于当前状态生成新报告, 自然能对比出哪些问题解决了 / 哪些仍存在。

## 采纳建议的流程

1. **review 报告** — 打开最新报告, 过一遍 🔴/🟡/🔵 三段
2. **分诊采纳** — 对每条建议:
   - 采纳 → 下一步
   - 不采纳 → 跳过 (下次报告会再提, 如果问题还在)
   - 存疑 → 开 GitHub issue 或团队讨论
3. **走正常流程固化**:
   | 建议类型 | 处理 |
   |---------|------|
   | 规则不一致 / 规则漂移 | 改 `.claude/rules/` |
   | 命令/skill/agent 问题 | 改对应 `.md` 文件 |
   | 代码违规 (硬编码等) | 走 `/fix` 或正常开发流程 |
   | 文档过期 | 改 `docs/` 或 `CLAUDE.md` |
4. **不要在报告里做标记** — 下次 `/meta-audit` 自然检测到变化并在「趋势 · 已解决」里体现

## 触发方式

### 手动

```bash
# 完整扫描
/meta-audit

# 只扫某维度
/meta-audit --focus=traceability
/meta-audit --focus=dead-links
```

### 建议频率

- **里程碑后**: 大版本 / 里程碑完成后跑一次, 看整体健康度
- **定期**: 1-2 周一次, 避免漂移累积
- **加入新机制后**: 加了新命令 / 规则 / agent 后跑一次, 看是否有一致性问题

**不推荐**: 自动化定时跑 — 报告没人看就是噪音, 手动触发能确保有人会读。

## 设计原则

- **观察而非裁决** — 报告不是绝对真理, 是基于当前规则的观察
- **人决策而非 agent 决策** — agent 不修任何东西, 所有改动过人脑
- **基于趋势而非单点** — 连续几次都报同一问题 = 真的要改; 只报一次可能是噪音
- **保留正反馈** — 报告也要列「做得好的地方」, 避免只看负面
